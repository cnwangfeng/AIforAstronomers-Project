\chapter{Unix/Linux 与科研计算环境}

\section{本章阅读方式}

本章快读命令说明，重点完成一次可复查的命令行操作记录。你不需要成为 Linux 专家，但必须能定位项目路径、预览写操作、保存日志、运行最小脚本，并说明输入和输出文件在哪里。

最低交付是一个 Evidence Record，加上一份命令日志或最小脚本。本章进入 Part III 的作用，是保证后续数据处理、训练脚本和模型输出都有清楚路径、日志和运行入口。

\section{本章导读：从会敲命令到会组织科研工作}

许多学生第一次学习 Linux 时，会把它理解成一张命令表：\texttt{ls} 是看文件，\texttt{cd} 是换目录，\texttt{cp} 是复制，\texttt{rm} 是删除。这样学当然能应付几道练习题，但一旦进入真实科研环境，问题马上会变得更复杂：服务器上有几千个文件，数据分散在多个目录，脚本要运行很久，输出需要保存，错误日志需要检查，别人还要知道你到底做了哪些操作。

因此，本章不把 Unix/Linux 当成一组孤立命令，而把它当成\textbf{科研计算工作流的最低层接口}。这一层接口负责回答五个问题：

\begin{enumerate}
    \item 我现在在哪个计算环境里？
    \item 我要处理的文件、目录和数据在哪里？
    \item 这条命令会读取什么、写入什么、改变什么？
    \item 这一步运行后留下了什么可检查的证据？
    \item 这套操作能否被自己、同学或助教重新执行？
\end{enumerate}

如果只会背命令，但不能回答这些问题，那么学生在远程服务器、共享目录或期末项目里仍然会迷路。本章的目标不是把所有 Linux 命令讲完，而是让你建立一个稳定的科研计算习惯：\textbf{先定位，再检查，再操作，再记录，最后自动化}。

\begin{defbox}[Unix/Linux]
Unix/Linux 是一类多用户、多任务操作系统及其工具生态。现代天文、物理和数据科学中的服务器、集群、远程计算环境、容器和许多科研软件，都大量使用 Unix/Linux 风格的命令行接口。
\end{defbox}

本章适合用“读一小节，马上做一小步”的方式学习。不要只看示例命令，也不要在自己的重要目录里随便试。最好的方法是在一个临时项目目录中创建少量模拟文件，然后反复练习定位、查看、搜索、复制、记录和归档。命令行不是靠眼睛学会的，它更像实验操作，必须动手。

\section{学习目标与训练目标}

学完本章后，你应该能够做到两类事情。第一类是\textbf{读懂}：能解释文件系统、路径、shell、命令、参数、重定向、管道、权限、进程和远程登录这些概念。第二类是\textbf{练会}：能在一个小型科研项目目录中完成最小的数据整理和运行记录。

具体来说，你应该能够：

\begin{itemize}
    \item 解释为什么 Unix/Linux 是现代天文与物理科研中的常见基础环境；
    \item 区分根目录、家目录、当前工作目录、绝对路径和相对路径；
    \item 使用 \texttt{pwd}、\texttt{ls}、\texttt{cd}、\texttt{mkdir}、\texttt{cp}、\texttt{mv}、\texttt{rm}、\texttt{find} 和 \texttt{grep}/\texttt{rg} 完成基本文件管理；
    \item 理解 shell 如何解析一条命令，包括命令名、选项、参数、通配符、引号和重定向；
    \item 使用 \texttt{head}、\texttt{tail}、\texttt{wc}、\texttt{sort}、\texttt{uniq}、\texttt{cut}、\texttt{xargs} 对文本文件和路径列表做快速检查；
    \item 使用 \texttt{which}、\texttt{type}、\texttt{PATH} 和环境变量检查命令来源；
    \item 把稳定的多步操作写成最小 shell 脚本，并保存运行日志；
    \item 在破坏性命令前完成预览、确认和备份。
\end{itemize}

训练材料的目标略有不同。配套 notebook 不负责替代正文解释，而负责让你\textbf{反复操作}。本章结束时，你至少应该交出一份小型运行记录，其中包含项目目录结构、关键命令、输出摘要、脚本入口、归档文件和已知风险。

\section{为什么科研工作离不开命令行}

图形界面很直观，也很适合低频、一次性的操作。例如打开一个文件、拖动一张图片、检查一个目录中的少量文件，这些事情用鼠标做没有问题。但科研计算经常具有另外几种特征。

\begin{itemize}
    \item \textbf{文件很多}：一次巡天观测可能产生大量图像、光谱、日志和中间结果。
    \item \textbf{任务重复}：同一段处理流程要应用到多个目标、多个夜晚或多个参数设置。
    \item \textbf{环境远程}：GPU 节点、HPC 集群、实验室服务器和云主机通常没有完整图形桌面。
    \item \textbf{过程要留痕}：科研结果需要解释“用什么数据、在哪个环境、运行了哪些步骤”。
\end{itemize}

命令行的优势正来自这些场景。你输入的命令既是操作，也是记录；你写出的脚本既是自动化工具，也是可复现说明。对于科研来说，命令行不是“更酷的打开文件方式”，而是一种把文件、程序、环境和证据连接起来的接口。

\begin{examplebox}[一个真实感很强的小问题]
假设一位学生拿到一夜观测后的目录，里面有图像文件、日志文件和上一位同学留下的脚本。老师问：
\begin{itemize}
    \item 一共有多少个 FITS 文件？
    \item 日志里有没有 \texttt{target042} 的记录？
    \item 哪些脚本最近被改动过？
    \item 这次运行的输出保存在哪里？
\end{itemize}
这些问题如果靠手动点击目录，既慢又容易漏。用命令行则可以把检查过程写下来，并在以后复查。
\end{examplebox}

这一章的核心思想是：\textbf{Linux 命令不是目的，科研工作流才是目的}。如果某条命令不能帮助你定位、检查、转换、记录或复现，它就不是本章的重点。

\section{Shell：你和操作系统之间的解释器}

终端窗口里真正和你对话的程序通常叫 \textbf{shell}。shell 读取你输入的文本，解析命令名、选项、参数、通配符、变量和重定向，然后把处理后的指令交给系统执行。

\begin{defbox}[Shell]
Shell 是命令解释器。它接收用户输入的命令文本，完成路径展开、变量替换、重定向、管道连接等预处理，然后调用真正的程序。
\end{defbox}

这一点很重要，因为很多错误并不是程序本身造成的，而是 shell 在程序运行前就已经把你的输入解释成了另一个含义。例如：

\begin{itemize}
    \item \texttt{*.fits} 会被 shell 展开成当前目录中匹配的文件名列表；
    \item \texttt{"*.fits"} 因为加了引号，通常不会被展开，而是作为普通字符串传给程序；
    \item \texttt{>} 会把标准输出写入文件，并可能覆盖已有文件；
    \item \texttt{|} 会把前一条命令的输出传给后一条命令。
\end{itemize}

所以学习 Linux 时，不能只问“这个命令是什么意思”，还要问“shell 在程序收到参数之前做了什么”。这就是命令行入门和科研安全之间的连接点。

\section{一条命令的解剖}

一条命令通常由三部分组成：命令名、选项和参数。以查看原始数据目录为例：

\begin{examplebox}[命令的基本结构]
\begin{lstlisting}[style=terminal]
$ ls -lh raw
\end{lstlisting}
\end{examplebox}

这里：

\begin{itemize}
    \item \texttt{ls} 是命令名，表示列出目录内容；
    \item \texttt{-l} 是选项，表示使用长格式；
    \item \texttt{-h} 是选项，表示用更容易读的单位显示大小；
    \item \texttt{raw} 是参数，表示检查对象是 \texttt{raw/} 目录；
    \item 输出默认显示在屏幕上，不会修改文件。
\end{itemize}

读一条命令时，可以固定问四个问题：

\begin{enumerate}
    \item \textbf{谁在执行？} 命令来自系统、Python 环境、Git，还是你自己写的脚本？
    \item \textbf{怎样执行？} 选项改变了什么行为？
    \item \textbf{作用于谁？} 参数指向哪个文件、目录、关键词或模式？
    \item \textbf{结果到哪里？} 输出到屏幕、写入文件、传给下一条命令，还是改变磁盘内容？
\end{enumerate}

这四问比死记命令更重要。它们可以帮助你结合表~\ref{tab:ch01_command_risk} 判断一条命令是只读检查，还是会改变项目状态。

\toprule

\midrule
只读检查 & \texttt{pwd}, \texttt{ls}, \texttt{head}, \texttt{rg} & 通常不改文件，适合先练习 \\
写入或移动 & \texttt{cp}, \texttt{mv}, \texttt{mkdir}, \texttt{>} & 会改变目录或文件，需要确认目标 \\

\section{帮助系统：把不会用变成会查}

初学命令行时，不可能也不应该记住所有选项。更可靠的学习方式是查帮助、读示例、只练当前任务需要的部分。

\begin{examplebox}[查看帮助]
\begin{lstlisting}[style=terminal]
$ ls --help
$ man ls
$ history
\end{lstlisting}
\end{examplebox}

\texttt{--help} 通常给出简要说明，\texttt{man} 提供更完整的手册页，\texttt{history} 可以查看自己曾经运行过的命令。帮助系统不是附录，而是命令行学习的主要入口。

例如你不确定 \texttt{ls -lh} 中 \texttt{-h} 的含义，可以运行：

\begin{examplebox}[在帮助中搜索关键词]
\begin{lstlisting}[style=terminal]
$ ls --help | grep human
\end{lstlisting}
\end{examplebox}

这条命令本身也展示了一个重要思想：帮助文本也是文本，可以搜索、筛选和保存。科研计算中的很多输入和输出最终都是文本，命令行擅长处理文本，正是它长期有效的原因之一。

\begin{warnbox}[不要复制自己不理解的命令]
从网页、聊天工具或 AI 助手那里复制命令前，至少要能说清楚它会读哪些文件、写哪些文件、是否包含 \texttt{rm}、\texttt{mv}、\texttt{>}、\texttt{chmod} 或远程传输。不能回答这些问题时，不要在真实数据目录中运行。
\end{warnbox}

\section{文件系统：所有操作都从路径开始}

Linux 文件系统是树状结构。所有路径都从根目录 \texttt{/} 开始向下展开。你日常工作的默认位置通常是家目录，例如 \texttt{/home/student} 或 macOS 上的 \texttt{/Users/student}。科研项目最好放在一个清楚命名的项目目录中，而不是散落在桌面、下载目录和临时文件夹里。

\begin{defbox}[路径]
路径是文件或目录在文件系统中的位置描述。绝对路径从根目录开始，例如 \texttt{/home/student/gaia\_night1/raw}；相对路径从当前工作目录开始，例如 \texttt{raw/night1\_target042.fits}。
\end{defbox}

路径理解不清，是命令行入门中最常见的错误源。很多“文件不存在”的报错，本质上不是文件真的不存在，而是你当前所在目录和你以为的不一样。

\begin{examplebox}[三个定位命令]
\begin{lstlisting}[style=terminal]
$ pwd
$ ls
$ ls raw
\end{lstlisting}
\end{examplebox}

\texttt{pwd} 回答“我在哪”，\texttt{ls} 回答“这里有什么”，\texttt{ls raw} 回答“这个子目录里有什么”。这三个问题构成了所有文件操作的起点。任何写操作之前，都应该先确认位置和对象。

\section{目录导航：不要在文件系统里迷路}

切换目录使用 \texttt{cd}。相对路径和绝对路径都可以作为参数：

\begin{examplebox}[切换目录]
\begin{lstlisting}[style=terminal]
$ cd gaia_night1
$ pwd
$ cd raw
$ pwd
$ cd ..
$ pwd
\end{lstlisting}
\end{examplebox}

这里的 \texttt{..} 表示上一级目录，\texttt{.} 表示当前目录。初学者需要特别小心的一点是：同样的相对路径，在不同当前目录下指向不同对象。例如在项目根目录中，下面这个路径是有效的：

\begin{lstlisting}[style=terminal]
raw/night1_target042.fits
\end{lstlisting}

但如果你已经进入 \texttt{raw/}，再写同样的相对路径，shell 会把它解释成：

\begin{lstlisting}[style=terminal]
raw/raw/night1_target042.fits
\end{lstlisting}

这通常会报错，因为当前位置下并不存在第二层 \texttt{raw/} 目录。

一个稳妥的训练习惯是：每次进入一个目录后，马上运行 \texttt{pwd} 和 \texttt{ls}。看似多余，实际上是在建立空间感。等你熟练之后可以减少频率，但初学阶段不要省这两步。

\section{查看目录内容：从文件名读出项目状态}

\texttt{ls} 不只是“看一下文件”。在科研项目中，它还帮助你判断目录是否整洁、文件是否按预期生成、时间戳是否合理、大小是否异常。

\begin{examplebox}[常用查看方式]
\begin{lstlisting}[style=terminal]
$ ls
$ ls -l
$ ls -lh
$ ls -lh raw logs results
$ ls -lt results
\end{lstlisting}
\end{examplebox}

\texttt{-l} 显示长格式，包含权限、所有者、大小、修改时间和文件名。\texttt{-h} 让文件大小更易读。\texttt{-t} 按修改时间排序，适合检查“刚才运行的结果是否真的更新了”。

看到 \texttt{ls -l} 输出时，不必一开始理解每个字段的全部细节，但至少要知道：

\begin{itemize}
    \item 第一列包含文件类型和权限；
    \item 中间列包含所有者、大小和修改时间；
    \item 最后一列是文件或目录名；
    \item 以 \texttt{d} 开头的是目录，以 \texttt{-} 开头的通常是普通文件。
\end{itemize}

这类观察会帮助你在后续章节中理解脚本权限、数据文件大小、输出更新时间和远程服务器权限问题。

\section{创建、复制、移动和删除}

项目开始时，常常需要创建目录结构。最小命令是 \texttt{mkdir}：

\begin{examplebox}[创建项目目录结构]
\begin{lstlisting}[style=terminal]
$ mkdir -p gaia_night1/raw gaia_night1/logs
$ mkdir -p gaia_night1/scripts gaia_night1/results gaia_night1/notes
\end{lstlisting}
\end{examplebox}

\texttt{-p} 的意思是：如果中间目录不存在就一起创建，如果目录已存在也不要报错。这对于脚本化创建项目结构很有用。

复制、移动和删除分别使用 \texttt{cp}、\texttt{mv}、\texttt{rm}：

\begin{examplebox}[复制、改名与删除]
\begin{lstlisting}[style=terminal]
$ cp raw/night1_target042.fits results/target042_copy.fits
$ mv results/target042_copy.fits results/target042_check.fits
$ rm results/target042_check.fits
\end{lstlisting}
\end{examplebox}

这三条命令的风险等级不同。\texttt{cp} 通常较安全，但可能覆盖同名目标。\texttt{mv} 既可以移动文件，也可以改名，如果目标写错，文件会出现在意外位置。\texttt{rm} 删除文件，通常不会进入图形界面的回收站。

\begin{warnbox}[\texttt{rm} 没有温柔的撤销按钮]
初学阶段不要在真实数据目录中练习 \texttt{rm -r} 或 \texttt{rm -rf}。如果必须删除目录，先用 \texttt{ls} 或 \texttt{find} 预览目标，再删除。任何批量删除都应该能解释目标集合来自哪里。
\end{warnbox}

\section{路径解析例题：先在脑中运行命令}

很多命令行错误不是因为命令不会用，而是因为路径没有在脑中先解析一遍。下面用同一个项目目录做一个小例题。假设你当前在项目根目录：

\begin{lstlisting}[style=terminal]
gaia_night1/
  raw/
  logs/
  results/
\end{lstlisting}

这时命令

\begin{lstlisting}[style=terminal]
$ cp raw/night1_target042.fits results/
\end{lstlisting}

可以读成一句完整的话：从当前目录下的 \texttt{raw/} 子目录读取 \texttt{night1\_target042.fits}，复制到当前目录下的 \texttt{results/} 子目录。若你先运行了

\begin{lstlisting}[style=terminal]
$ cd raw
\end{lstlisting}

同样的相对路径就不再指向原文件，而会被解释成下面这个新路径：

\begin{lstlisting}[style=terminal]
raw/raw/night1_target042.fits
\end{lstlisting}

这就是为什么写命令前要先问：\textbf{当前工作目录是什么，路径是从哪里开始算的。}

换成自然语言，这个错误其实很容易发现：你已经站在 \texttt{raw/} 里面，却又要求系统“再进入一个名叫 \texttt{raw/} 的子目录”。如果当前目录下没有第二层 \texttt{raw/}，命令就会失败；如果恰好有同名子目录，命令甚至可能作用到完全不是你想要的文件。路径错误最危险的地方不一定是报错，而是\textbf{没有报错但作用对象错了}。

\begin{protip}[把相对路径翻译成自然语言]
每次遇到复杂路径时，先把它翻译成“从哪里出发、进入哪个目录、作用于哪个文件”。如果这句话说不清，就不要执行写操作。
\end{protip}

\section{通配符与引号：shell 先帮你选文件}

通配符让你用模式表示一组文件。常见模式包括：

\begin{itemize}
    \item \texttt{*.fits}：匹配当前目录下所有以 \texttt{.fits} 结尾的文件；
    \item \texttt{night1\_*.fits}：匹配所有以 \texttt{night1\_} 开头、以 \texttt{.fits} 结尾的文件；
    \item \texttt{target??.fits}：用两个问号匹配两个任意字符。
\end{itemize}

\begin{examplebox}[先预览再批量操作]
\begin{lstlisting}[style=terminal]
$ ls raw/*.fits
$ cp raw/*.fits results/
\end{lstlisting}
\end{examplebox}

这里真正执行 \texttt{cp} 之前，shell 会先把 \texttt{raw/*.fits} 展开成具体文件名列表。因此，批量复制、移动或删除之前，最好先用 \texttt{ls} 预览匹配结果。

引号会改变 shell 的解释方式：

\begin{examplebox}[引号改变展开行为]
\begin{lstlisting}[style=terminal]
$ echo raw/*.fits
$ echo "raw/*.fits"
\end{lstlisting}
\end{examplebox}

第一条命令通常输出匹配到的文件名，第二条命令输出字面字符串 \texttt{raw/*.fits}。这解释了为什么很多命令在处理包含空格、星号或特殊字符的路径时需要引号。

\section{查看文件内容：先抽样，不要一上来全打开}

科研日志、CSV 表格和配置文件通常是文本文件。查看文本时，不要一开始就把完整文件打印到屏幕，尤其是文件很大时。更好的方法是先抽样。

\begin{examplebox}[文本文件快速预览]
\begin{lstlisting}[style=terminal]
$ head logs/night1.log
$ tail logs/night1.log
$ wc -l logs/night1.log
$ less logs/night1.log
\end{lstlisting}
\end{examplebox}

\texttt{head} 看开头，\texttt{tail} 看结尾，\texttt{wc -l} 统计行数，\texttt{less} 分页浏览。对于大型日志，\texttt{tail} 尤其重要，因为错误信息常常出现在最后。

对于表格文件，第一眼应该检查三件事：

\begin{itemize}
    \item 是否有表头；
    \item 分隔符是什么，例如逗号、空格还是制表符；
    \item 行数是否接近预期。
\end{itemize}

这三件事看似简单，却能避免许多后续 Python 读入错误。

\section{搜索：让计算机替你翻目录}

当文件很多时，搜索比肉眼翻目录重要得多。搜索有两类常见对象：搜索文件名和搜索文件内容。

\begin{examplebox}[按文件名搜索]
\begin{lstlisting}[style=terminal]
$ find raw -name "*.fits"
$ find . -name "*.log"
\end{lstlisting}
\end{examplebox}

\texttt{find} 的基本思想是：从某个目录开始，按条件查找路径。上面的 \texttt{find raw -name "*.fits"} 表示从 \texttt{raw/} 开始，寻找名字匹配 \texttt{*.fits} 的文件。

搜索文件内容可以用 \texttt{grep} 或更现代的 \texttt{rg}：

\begin{examplebox}[按内容搜索]
\begin{lstlisting}[style=terminal]
$ grep "target042" logs/night1.log
$ grep -R "ERROR" logs
$ rg "target042|ERROR" logs
\end{lstlisting}
\end{examplebox}

\texttt{grep -R} 递归搜索目录，\texttt{rg} 通常速度更快、默认行为更适合项目目录。课程中如果系统没有安装 \texttt{rg}，使用 \texttt{grep} 即可。

\section{快速统计：把输出变成检查表}

科研项目中常常需要快速回答“有多少”“有哪些”“是否重复”。这些问题不一定需要 Python，很多时候终端工具已经足够。

\begin{examplebox}[文件数量和后缀统计]
\begin{lstlisting}[style=terminal]
$ find raw -name "*.fits" | wc -l
$ find raw -type f | sed 's/.*\\.//' | sort | uniq -c
\end{lstlisting}
\end{examplebox}

第一条命令统计 FITS 文件数量。第二条命令从文件名中提取扩展名，排序后统计每种扩展名出现次数。即使你还不熟悉 \texttt{sed}，也应该看懂这条管道的思路：\textbf{列出文件 -> 提取字段 -> 排序 -> 统计}。

更常见的文本字段检查可以用 \texttt{cut}、\texttt{sort} 和 \texttt{uniq}：

\begin{examplebox}[从 CSV 中统计类别]
\begin{lstlisting}[style=terminal]
$ cut -d, -f3 targets.csv | sort | uniq -c
\end{lstlisting}
\end{examplebox}

这条命令假设 \texttt{targets.csv} 是逗号分隔文件，第三列是类别。它不是严肃数据分析的终点，但非常适合做第一轮 sanity check。

\section{重定向：把屏幕输出保存成证据}

默认情况下，许多命令把结果打印到屏幕。屏幕输出看完就消失，不适合当科研记录。重定向可以把输出写入文件。

\begin{examplebox}[保存目录快照]
\begin{lstlisting}[style=terminal]
$ ls -lh raw > logs/raw_listing.txt
$ find raw -name "*.fits" | sort > logs/fits_files.txt
\end{lstlisting}
\end{examplebox}

\texttt{>} 会覆盖目标文件，\texttt{>>} 会追加到目标文件末尾：

\begin{examplebox}[覆盖与追加]
\begin{lstlisting}[style=terminal]
$ date > logs/run_record.txt
$ pwd >> logs/run_record.txt
$ find raw -name "*.fits" | wc -l >> logs/run_record.txt
\end{lstlisting}
\end{examplebox}

覆盖和追加的差异非常重要。很多学生以为自己在“保存一份记录”，其实用 \texttt{>} 覆盖了旧记录。写入前先确认目标文件是否已经存在，是一个很好的习惯。

\section{管道：把小工具接成流程}

管道符号 \texttt{|} 会把前一条命令的标准输出传给后一条命令的标准输入。它让多个小工具组合成一个工作流。

\begin{examplebox}[管道示例]
\begin{lstlisting}[style=terminal]
$ find raw -name "*.fits" | sort | head
$ grep "ERROR" logs/night1.log | wc -l
$ find raw -type f | sort | tee logs/all_raw_files.txt | wc -l
\end{lstlisting}
\end{examplebox}

第三条命令中的 \texttt{tee} 很适合科研记录：它一边把输出继续传给后面的 \texttt{wc -l}，一边保存一份到 \texttt{logs/all\_raw\_files.txt}。这样你既能看到数量，也能保留文件清单。

管道的思维方式可以概括为：

\begin{center}
\texttt{产生列表 -> 筛选 -> 排序 -> 统计 -> 保存证据}
\end{center}

后续学习 Python、pandas、机器学习和深度学习时，这种“数据流”思维仍然会反复出现。

\section{管道 worked example：从科研问题到命令}

假设你想回答一个很普通但真实的问题：\textbf{这夜观测中，名字里带 target 的 FITS 文件有多少个，清单是什么？}

不要一开始就写复杂脚本。可以先把问题拆成四步：

\begin{enumerate}
    \item 从 \texttt{raw/} 中列出所有 FITS 文件；
    \item 只保留名字中包含 \texttt{target} 的文件；
    \item 排序，保证记录稳定；
    \item 一边保存清单，一边统计数量。
\end{enumerate}

对应的命令是：

\begin{examplebox}[从问题到管道]
\begin{lstlisting}[style=terminal]
$ find raw -name "*.fits" \
    | grep "target" \
    | sort \
    | tee logs/target_fits_files.txt \
    | wc -l
\end{lstlisting}
\end{examplebox}

这条命令的价值不只是得到一个数字。它还留下了 \texttt{logs/target\_fits\_files.txt}，以后你可以检查到底统计了哪些文件。如果只保存最后的数量，别人就无法判断这个数量是否来自正确的筛选条件。

\begin{warnbox}[管道中的每一步都应该能单独解释]
如果一条管道很长，不要一次性相信它。先运行前一段，例如只运行到 \texttt{grep "target"}，确认输出符合预期，再继续接 \texttt{sort}、\texttt{tee} 和 \texttt{wc -l}。
\end{warnbox}

\section{标准输出、标准错误和退出状态}

命令运行时通常有三条信息通道：

\begin{itemize}
    \item 标准输入：程序读取的数据来源；
    \item 标准输出：正常结果；
    \item 标准错误：错误、警告和诊断信息。
\end{itemize}

有些程序即使出错，也可能仍然把部分正常输出打印出来。因此，长任务最好同时保存输出和错误。

\begin{examplebox}[保存输出和错误]
\begin{lstlisting}[style=terminal]
$ ./scripts/run_pipeline.sh > logs/pipeline.out 2> logs/pipeline.err
$ echo $?
\end{lstlisting}
\end{examplebox}

\texttt{\$?} 保存上一条命令的退出状态。通常 \texttt{0} 表示成功，非零值表示失败。脚本中可以利用退出状态判断是否继续执行后续步骤。

\begin{protip}[错误日志不是垃圾]
科研项目中的 \texttt{.err}、\texttt{.log} 和运行记录常常比最终图更能说明问题来源。不要在没有检查的情况下删除它们。
\end{protip}

\section{命令从哪里来：\texttt{PATH}、环境变量与 alias}

当你输入 \texttt{python}、\texttt{git} 或 \texttt{pip} 时，shell 需要知道去哪里找对应程序。这个查找路径由环境变量 \texttt{PATH} 控制。

\begin{examplebox}[检查命令来源]
\begin{lstlisting}[style=terminal]
$ which python
$ python --version
$ which git
$ type cd
$ echo $PATH
\end{lstlisting}
\end{examplebox}

\texttt{which} 告诉你外部命令来自哪个路径，\texttt{type} 还能识别 shell 内置命令和 alias。科研计算中，这一步非常关键：同样输入 \texttt{python}，不同 conda 环境、不同服务器、不同终端配置可能运行的是不同解释器。

环境变量可以保存路径或配置：

\begin{examplebox}[设置项目路径变量]
\begin{lstlisting}[style=terminal]
$ export PROJECT_ROOT=$HOME/gaia_night1
$ echo $PROJECT_ROOT
\end{lstlisting}
\end{examplebox}

alias 可以给常用命令起短名字：

\begin{examplebox}[一个常见 alias]
\begin{lstlisting}[style=terminal]
$ alias ll='ls -lh'
\end{lstlisting}
\end{examplebox}

环境变量和 alias 很方便，但要有边界。脚本中不要依赖只有你自己终端里才有的 alias；需要依赖的环境变量应该写进 README 或脚本说明。否则流程只能在你自己的电脑上“刚好能跑”，不能成为可复现科研工作流。

\section{权限与可执行脚本}

Linux 中每个文件都有权限。使用 \texttt{ls -l} 时，第一列类似：

\begin{lstlisting}[style=terminal]
-rw-r--r--
drwxr-xr-x
\end{lstlisting}

第一位表示类型，\texttt{-} 通常是普通文件，\texttt{d} 是目录。后面的字符表示所有者、用户组和其他用户的读、写、执行权限。

脚本要直接运行，通常需要执行权限：

\begin{examplebox}[给脚本增加执行权限]
\begin{lstlisting}[style=terminal]
$ ls -l scripts/run_pipeline.sh
$ chmod u+x scripts/run_pipeline.sh
$ ./scripts/run_pipeline.sh
\end{lstlisting}
\end{examplebox}

这里的 \texttt{u+x} 表示给文件所有者增加执行权限。执行脚本前面的 \texttt{./} 表示运行当前目录下的这个脚本，而不是去 \texttt{PATH} 中寻找同名命令。

\section{权限数字：\texttt{chmod 755} 为什么是 755}

你也会看到下面这种写法：

\begin{lstlisting}[style=terminal]
$ chmod 755 scripts/run_pipeline.sh
\end{lstlisting}

这里的三个数字分别对应所有者、用户组和其他用户。每个数字由读、写、执行三种权限相加得到：

\begin{center}
\begin{tabular}{ccc}
\toprule
权限 & 数值 & 含义 \\
\midrule
r & 4 & read，读取 \\
w & 2 & write，写入或修改 \\
x & 1 & execute，执行或进入目录 \\
\bottomrule
\end{tabular}
\end{center}

所以 \texttt{755} 表示：

\[
7=4+2+1,\qquad 5=4+1,\qquad 5=4+1.
\]

也就是所有者可读、可写、可执行；用户组和其他用户可读、可执行但不可写。入门阶段不需要记住所有组合，但要理解：权限不是装饰，它决定了谁能读取数据、谁能修改脚本、谁能运行入口。共享服务器上随意给写权限，可能会让别人意外修改你的项目；权限过紧，又可能让助教或脚本无法读取必要文件。

\section{远程服务器：同一套习惯换一个环境}

真实科研中，你很可能需要登录远程服务器或集群节点。最常见的入口是 \texttt{ssh}：

\begin{examplebox}[远程登录与传输]
\begin{lstlisting}[style=terminal]
$ ssh student@example.edu
$ scp local_file.txt student@example.edu:/home/student/project/
$ scp student@example.edu:/home/student/project/results/summary.txt .
\end{lstlisting}
\end{examplebox}

远程环境的核心困难不是 \texttt{ssh} 命令本身，而是上下文切换。你需要确认：

\begin{itemize}
    \item 现在是在本地终端还是远程服务器；
    \item 当前目录在哪里；
    \item 使用的是远程环境中的哪个 \texttt{python}；
    \item 数据路径是本地路径还是服务器路径；
    \item 传输方向是本地到远程，还是远程到本地。
\end{itemize}

\begin{warnbox}[远程路径不要靠猜]
\texttt{scp} 的两端路径都要写清楚。初学者常见错误是把本地路径和远程路径混在一起，或者把传输方向写反。传输前先用 \texttt{pwd} 和 \texttt{ls} 分别确认两端位置。
\end{warnbox}

\section{远程运行的最小安全流程}

在远程服务器上运行脚本前，建议固定做一次小检查：

\begin{examplebox}[远程运行前检查]
\begin{lstlisting}[style=terminal]
$ hostname
$ pwd
$ which python
$ python --version
$ git status --short
$ ls data
\end{lstlisting}
\end{examplebox}

这些命令分别回答：我在哪台机器、在哪个目录、用哪个 Python、代码是否有未提交改动、数据是否在预期位置。它们看起来琐碎，但能避免非常多的远程环境错误。尤其在集群、教学服务器或多 conda 环境中，\texttt{which python} 往往比你以为的重要。

如果任务要长时间运行，应把命令、环境和日志位置写入记录，而不是只在聊天中说“我在服务器上跑了”。远程运行更需要证据，因为你离开终端后，很多上下文会消失。

\section{进程与长任务：运行不是瞬间完成的}

许多科研任务不是一秒钟结束。模型训练、批量图像处理、光谱拟合和大型下载都可能运行很久。你需要知道任务是否还在运行、是否失败、输出在哪里。

\begin{examplebox}[查看进程和运行时间]
\begin{lstlisting}[style=terminal]
$ ps
$ top
$ time ./scripts/run_pipeline.sh
\end{lstlisting}
\end{examplebox}

对于长任务，最小运行记录应该包含：

\begin{itemize}
    \item 启动时间；
    \item 当前目录；
    \item 运行命令；
    \item 标准输出和错误日志位置；
    \item 退出状态；
    \item 关键输入文件和输出文件。
\end{itemize}

\begin{examplebox}[长任务的最小记录方式]
\begin{lstlisting}[style=terminal]
$ date > logs/pipeline_run.txt
$ pwd >> logs/pipeline_run.txt
$ ./scripts/run_pipeline.sh > logs/pipeline.out 2> logs/pipeline.err
$ echo $? >> logs/pipeline_run.txt
\end{lstlisting}
\end{examplebox}

这不是繁文缛节。几天后当你发现结果异常，或者同学问你“这张图怎么生成的”，这些记录会节省大量时间。

\section{最小 shell 脚本：把稳定操作变成入口}

当你发现自己反复运行同一串命令时，就应该考虑写脚本。脚本不是为了炫技，而是为了把手工步骤变成可检查、可复用、可复现的入口。

\begin{examplebox}[最小 shell 脚本]
\begin{lstlisting}[style=terminal]
#!/usr/bin/env bash
set -euo pipefail

mkdir -p logs results
date > logs/run_pipeline_record.txt
pwd >> logs/run_pipeline_record.txt
find raw -name "*.fits" | sort > logs/input_fits_files.txt
echo "pipeline-ready" > results/status.txt
\end{lstlisting}
\end{examplebox}

第一行称为 shebang，说明用哪个解释器运行脚本。\texttt{set -euo pipefail} 是一种常见安全设置：命令失败、使用未定义变量或管道中间失败时，让脚本更早暴露问题。入门阶段可以先把它当成“不要悄悄失败”的保护。

一个适合科研项目的最小脚本，应该做到：

\begin{itemize}
    \item 从项目根目录运行；
    \item 明确创建输出目录；
    \item 保存关键输入清单；
    \item 把输出写入稳定位置；
    \item 失败时不要悄悄继续。
\end{itemize}

\section{归档：阶段性冻结项目状态}

当一个阶段完成时，最好把关键材料打包。归档不等于备份所有东西，而是把一组需要一起解释的文件冻结在一起。

\begin{examplebox}[创建和检查归档]
\begin{lstlisting}[style=terminal]
$ tar -czf results/night1_minimal_package.tar.gz scripts logs results/status.txt
$ tar -tzf results/night1_minimal_package.tar.gz | head
\end{lstlisting}
\end{examplebox}

\texttt{tar -czf} 创建压缩归档，\texttt{tar -tzf} 查看归档内容。归档后一定要检查里面有什么。很多人以为自己打包了结果，实际漏掉脚本或日志；也有人不小心把巨大原始数据一起打进包里。

在课程项目中，一个最小归档包通常包括脚本、运行记录、关键输出和 README。大型原始数据可以不进包，但必须说明来源和重新获取方式。

\section{破坏性命令前的三步确认}

Linux 命令行强大，也意味着它不会频繁打断你问“你确定吗”。因此，执行破坏性命令前必须形成固定流程。

\begin{enumerate}
    \item \textbf{预览目标}：用 \texttt{ls}、\texttt{find} 或 \texttt{echo} 查看将要作用的文件；
    \item \textbf{缩小范围}：优先指定目录和模式，不要在不确定位置使用宽泛通配符；
    \item \textbf{保留证据}：重要操作前保存文件清单，必要时先归档。
\end{enumerate}

\begin{examplebox}[删除前先预览]
\begin{lstlisting}[style=terminal]
$ find results -name "*.tmp" | sort
$ find results -name "*.tmp" | wc -l
$ find results -name "*.tmp" -delete
\end{lstlisting}
\end{examplebox}

第三条命令只有在前两条输出符合预期时才应该运行。对初学者来说，这种“多敲两条命令”的习惯比任何高级技巧都重要。

\section{从命令到科研工作流：最小算法}

把本章所有内容压缩成一条科研计算算法，可以写成：

\begin{enumerate}
    \item 创建清楚的项目目录；
    \item 用 \texttt{pwd}、\texttt{ls}、\texttt{find} 确认位置和输入；
    \item 用 \texttt{head}、\texttt{tail}、\texttt{wc}、\texttt{grep}/\texttt{rg} 做快速检查；
    \item 用重定向和 \texttt{tee} 保存关键输出；
    \item 把稳定步骤写进脚本；
    \item 用日志记录长任务；
    \item 阶段完成后打包关键材料；
    \item 删除或覆盖前先预览目标。
\end{enumerate}

这条算法比任何单条命令都重要。后续章节中的 Git、Python 脚本、数据 I/O、绘图、机器学习训练和 capstone 项目，都会依赖这条基础工作流。

\section{最小命令行 Evidence Record}

本章训练结束时，建议学生提交一份很短的 Evidence Record。它不需要漂亮，但应该能让别人知道你做过什么。

\begin{examplebox}[运行记录模板]
\begin{lstlisting}[style=terminal]
Project: gaia_night1
Date:
Machine or environment:
Working directory:

Input check:
- raw file count:
- log file checked:
- suspicious messages:

Commands run:
- mkdir -p ...
- find raw -name "*.fits" | sort > logs/input_fits_files.txt
- ./scripts/run_pipeline.sh > logs/pipeline.out 2> logs/pipeline.err

Outputs:
- results/status.txt
- logs/input_fits_files.txt
- logs/pipeline.out
- logs/pipeline.err

Known limits:
- simulated files only / real FITS headers not checked
- no scientific calibration yet
\end{lstlisting}
\end{examplebox}

这份记录使用统一 Evidence Record 母模板，本章只补充以下命令行字段：

\begin{examplebox}[Ch01 Evidence Record 附加字段]
\begin{lstlisting}[style=terminal]
Ch01 Fields:
- project root:
- key read-only checks:
- preview before write:
- log file:
- script or command entry:
- python path/version check:
\end{lstlisting}
\end{examplebox}

本章的特殊通过条件是：必须包含一次写操作前的 preview，或明确说明本章操作全部为只读。这份模板会在后续章节中逐步升级：Git 章会补上 commit hash，数据 I/O 章会补上字段契约，绘图章会补上图表说明，机器学习章会补上模型评估，最终进入 capstone evidence cards。

\section{配套 notebook 如何训练}

本章配套 notebook 不会假装在浏览器里复刻完整终端，而是在临时目录中创建一个小型项目，然后用 Python 调用安全的 shell 命令。它训练的是下面这条链：

\begin{enumerate}
    \item 创建微型项目目录；
    \item 检查当前位置和目录内容；
    \item 复制、移动和删除安全的模拟文件；
    \item 搜索日志中的关键词；
    \item 保存目录快照和文件清单；
    \item 修改脚本权限并运行最小脚本；
    \item 打包一个阶段性归档；
    \item 生成一份 chapter snapshot。
\end{enumerate}

正文负责讲清楚“为什么这样做”和“这些命令在科研中对应什么判断”；notebook 负责让你动手练到熟。读者不应该把 notebook 输出当作唯一答案，而应该能解释每一步输出和项目目录状态之间的关系。

\section{常见错误与诊断}

\begin{warnbox}[Linux 入门中的典型问题]
\begin{itemize}
    \item 只背命令，不确认当前目录，导致文件写到意外位置；
    \item 把原始数据、脚本、结果和日志混在同一个目录；
    \item 在真实数据目录中直接试 \texttt{rm}、批量 \texttt{mv} 或覆盖重定向；
    \item 复制网页命令但不理解通配符、引号和重定向；
    \item 不检查 \texttt{python}、\texttt{pip} 或 \texttt{git} 来自哪个环境；
    \item 长任务没有保存标准输出、标准错误和退出状态；
    \item 只保留最终结果，不保留输入清单和运行记录。
\end{itemize}
\end{warnbox}

遇到问题时，不要急着重跑。先回到三件事：当前位置、目标文件、最近输出。大部分入门错误都可以通过 \texttt{pwd}、\texttt{ls -lh}、\texttt{history}、日志文件和退出状态定位。

\section{AI 助手如何帮助你}

AI 助手可以帮助你解释命令、生成脚本草稿、阅读错误日志、整理运行记录和设计安全检查。但 AI 助手不应该替你跳过判断。尤其是包含删除、覆盖、权限修改、远程传输和批量处理的命令，必须先让它解释风险，再在临时目录中试运行。

比较好的提问方式是：

\begin{lstlisting}[style=terminal]
请解释这条命令每一部分的作用，并指出它会读哪些文件、
写哪些文件、是否可能覆盖或删除内容：
find results -name "*.tmp" -delete
\end{lstlisting}

更好的提问方式是要求 AI 给出预览版本：

\begin{lstlisting}[style=terminal]
我想删除 results/ 下的临时文件。请先给出只预览目标、
不删除文件的命令，再给出真正删除命令，并解释风险。
\end{lstlisting}

这类用法符合本章原则：AI 可以加速理解和检查，但最终运行命令的人必须知道命令的作用范围和证据位置。

\section{练习与交付}

\begin{exercisebox}[基础练习]
在临时目录中创建 \texttt{gaia\_night1/} 项目结构，包括 \texttt{raw/}、\texttt{logs/}、\texttt{scripts/}、\texttt{results/} 和 \texttt{notes/}。运行 \texttt{pwd}、\texttt{find . -maxdepth 2 -type d}，并把目录结构保存到 \texttt{logs/tree\_snapshot.txt}。
\end{exercisebox}

\begin{exercisebox}[过程练习]
创建三个模拟 \texttt{.fits} 文件和一个 \texttt{night1.log} 文件。用 \texttt{find} 统计 FITS 文件数量，用 \texttt{grep} 或 \texttt{rg} 搜索一个目标名，把文件清单和搜索结果保存到 \texttt{logs/}。
\end{exercisebox}

\begin{exercisebox}[诊断练习]
故意在错误目录中运行一次只读命令，例如 \texttt{ls raw}。解释错误信息说明了什么，然后用 \texttt{pwd} 和 \texttt{ls} 找回正确位置。不要用删除命令制造错误。
\end{exercisebox}

\begin{exercisebox}[交付练习]
写一个 \texttt{scripts/run\_pipeline.sh}，至少完成创建输出目录、保存输入文件清单、写入一个结果状态文件。运行脚本时把标准输出和标准错误分别保存到 \texttt{logs/}。最后提交一份最小命令行运行记录。
\end{exercisebox}

\section{Exit Gate}

\begin{enumerate}
    \item 我能否从项目根目录定位输入和输出？
    \item 我是否在写操作前做了 preview？
    \item 我是否保存了关键命令或日志？
    \item 我是否能运行一个最小脚本并检查 Python 路径/版本？
\end{enumerate}

最低交付：Evidence Record 和命令日志或最小脚本。

\section{本章小结}

Unix/Linux 入门不应该停留在“会敲几个命令”。对于天文与物理本科生来说，本章真正要建立的是科研计算底层习惯：清楚目录结构，知道路径含义，能区分只读检查和写操作，理解 shell 如何解释命令，能够搜索和统计文本输出，愿意保存日志和运行记录，并能把稳定步骤写成脚本。只要这套习惯建立起来，后面的 Git、Python、FITS、绘图、机器学习和 capstone 项目都会更稳。
