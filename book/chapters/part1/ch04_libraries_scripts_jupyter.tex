\chapter{库、脚本与 Jupyter}

\section{本章阅读方式}

本章需要精读并动手。目标不是多写 Python，而是把探索代码整理成可重跑入口：别人知道用哪个环境、运行哪个脚本、输入在哪里、输出到哪里、随机种子和参数是什么。

最低交付是一个 Evidence Record、一段能从命令行运行的脚本和一份运行日志。本章进入 Part III 的作用，是为机器学习实验提供可复查的环境、入口、参数、随机种子和输出目录。

\section{本章导读：从会写代码到会交付工作流}

学完 Python 基础语法之后，很多学生会自然进入 Jupyter notebook：一格一格运行代码，马上看到表格、数字和图像。这是非常好的探索方式。但真实科研项目不能停留在“这一页 notebook 在我电脑上刚才跑通了”。别人需要知道你用了哪些库、在哪个环境里运行、随机过程是否固定、哪些单元是临时探索、哪些脚本能从头重跑、最终结果从哪里生成。

本章讨论的不是更复杂的 Python 语法，而是科研编程中的一个分水岭：\textbf{代码能运行} 和 \textbf{工作流可复现} 是两件不同的事。前者只说明当前环境下某段代码没有立刻报错；后者要求你能解释输入、依赖、执行顺序、输出位置和验证记录。

在 Part I 的主线中，本章承接前两章：

\begin{itemize}
    \item 第 1 章教你把文件、目录、脚本和日志组织在一个 Linux 工作流里；
    \item 第 2 章教你把项目变化记录进 Git 时间轴；
    \item 本章要把 Python 代码放进这两个框架中，形成可运行、可检查、可维护的科研小项目。
\end{itemize}

如果说第 1 章训练的是“在哪里运行”，第 2 章训练的是“如何记录变化”，那么本章训练的是“如何把探索性代码变成别人也能运行的项目入口”。

\section{学习目标与训练目标}

学完本章后，你应该能够：

\begin{itemize}
    \item 区分标准库、第三方库、模块、脚本和 notebook 的不同角色；
    \item 解释为什么 \texttt{import}、环境文件、随机种子和路径配置都是科研记录的一部分；
    \item 判断一段代码适合留在 notebook、整理成 script，还是抽成 module；
    \item 使用 \texttt{venv} 或等价工具建立项目隔离环境，并记录关键依赖；
    \item 把探索性 notebook 中的稳定逻辑逐步整理成可重复运行的脚本；
    \item 用 \texttt{main()}、配置对象、日志和错误处理构造最小脚本入口；
    \item 写出一份最小脚本运行记录，说明输入、命令、环境、输出和限制。
\end{itemize}

配套训练的目标更具体：你要从一个小 notebook 式探索开始，把其中稳定部分整理成脚本，把复用函数放进 \texttt{src/} 或一个小模块，并交付一份运行记录。正文负责讲清楚为什么这样组织；notebook 负责让你亲手完成这次转换。

\section{库：复用别人已经验证过的工具}

Python 库的价值不只是“少写几行代码”。在科研中，库代表了经过大量用户使用、测试和修正的实现。你当然可以自己写求平均、读 CSV、画图和拟合函数，但这样做会把精力从科学问题转移到低层工具重造上。

\begin{defbox}[库]
库是一组已经组织好的函数、类和工具。标准库随 Python 一起安装，第三方库需要额外安装，科研项目通常同时依赖两者。
\end{defbox}

常见标准库包括：

\begin{itemize}
    \item \texttt{pathlib}：路径处理；
    \item \texttt{csv}：简单 CSV 读写；
    \item \texttt{math}：基础数学函数；
    \item \texttt{statistics}：简单统计量；
    \item \texttt{random}：随机数；
    \item \texttt{logging}：日志；
    \item \texttt{argparse}：命令行参数。
\end{itemize}

常见第三方科学库包括：

\begin{itemize}
    \item \texttt{numpy}：数组和向量化计算；
    \item \texttt{matplotlib}：绘图；
    \item \texttt{pandas}：表格数据；
    \item \texttt{astropy}：FITS、单位、坐标和时间；
    \item \texttt{scikit-learn}：机器学习 workflow；
    \item \texttt{torch}：深度学习和自动求导。
\end{itemize}

\begin{protip}[先问“有没有可信工具”，再问“我怎么重写”]
本科科研训练当然需要理解算法，但不等于每个底层工具都要自己实现。更成熟的习惯是：先理解问题，再选择可信库，最后检查库函数的输入、输出和假设。
\end{protip}

\section{\texttt{import} 是依赖声明}

很多初学者把 \texttt{import} 当成固定开头。其实它更像一份依赖声明：这段分析依赖哪些模块，运行环境中必须有什么，别人复现时需要安装什么。

\begin{examplebox}[一个很小但完整的依赖片段]
\begin{lstlisting}[style=pythonstyle]
from pathlib import Path
import csv
import statistics

DATA_PATH = Path("data/lightcurve_demo.csv")

with DATA_PATH.open(newline="", encoding="utf-8") as handle:
    rows = list(csv.DictReader(handle))

magnitudes = [float(row["mag"]) for row in rows if row["quality"] == "good"]
print(statistics.mean(magnitudes))
\end{lstlisting}
\end{examplebox}

这段代码已经包含若干科研判断：

\begin{itemize}
    \item 输入路径是项目相对路径，不是个人电脑上的绝对路径；
    \item 文件编码和 CSV 读取方式是显式的；
    \item 质量筛选条件写在代码中，可以被审查；
    \item 输出来自一条清楚的计算链。
\end{itemize}

如果一个 notebook 第一格只有大量导入，但没有说明为什么需要这些库，读者很难判断哪些依赖是真正必要的，哪些是早期试验留下的残留。

\section{安装、导入与依赖文件不是一回事}

初学者常把三句话混在一起：安装了库、导入了库、记录了依赖。它们相关，但不是同一件事，区别见表~\ref{tab:ch04_install_import_dependency}。

\begin{table}[htbp]
\centering
\begin{tabular}{p{0.20\textwidth}p{0.32\textwidth}p{0.36\textwidth}}
\toprule
动作 & 发生在哪里 & 它说明什么 \\
\midrule
安装 & 当前 Python 环境 & 这个环境里有某个包，但不代表项目记录了它 \\
导入 & 当前脚本或 notebook & 这段代码运行时需要某个模块 \\
记录依赖 & \texttt{requirements.txt} 或环境文件 & 别人有线索重建运行环境 \\
\bottomrule
\end{tabular}
\caption{安装、导入与依赖记录在科研 Python 项目中的区别。}
\label{tab:ch04_install_import_dependency}
\end{table}

例如你在终端中运行：

\begin{lstlisting}[style=terminal]
$ python -m pip install astropy
\end{lstlisting}

这只是改变了当前环境。只有当代码中出现 \texttt{from astropy.io import fits} 时，项目才真正使用了 \texttt{astropy}；只有当 \texttt{requirements.txt}、\texttt{environment.yml} 或 README 记录了这个依赖时，别人才能知道复现时需要安装它。

一个实用原则是：\textbf{能导入不等于可复现，能运行不等于已记录}。每次你为项目安装新库，都应该问：它是否真的被代码使用？如果使用了，是否写进依赖记录？如果只是一次临时探索，是否应该从最终依赖列表中清理掉？

\section{环境：同一段代码为什么会跑出不同结果}

一段科研分析可以抽象成：

\[
\mathrm{output}=f(\mathrm{input}, \mathrm{code}, \mathrm{parameters}, \mathrm{environment}, \mathrm{random\ state}).
\]

学生常常只盯着 \texttt{code}，但环境和随机状态同样可能改变结果。例如库版本不同、路径不同、默认数值精度不同、随机数没有固定，都会让“同一段代码”不再真的相同。

\begin{examplebox}[检查关键环境信息]
\begin{lstlisting}[style=terminal]
$ python --version
$ python -m pip freeze | head
$ which python
\end{lstlisting}
\end{examplebox}

在 Python 中也可以检查模块是否存在：

\begin{examplebox}[检查模块可用性]
\begin{lstlisting}[style=pythonstyle]
from importlib.util import find_spec

for name in ["numpy", "matplotlib", "astropy", "pandas"]:
    print(name, find_spec(name) is not None)
\end{lstlisting}
\end{examplebox}

这类检查不“高级”，但很有效。许多项目无法复现，不是因为科学思想错了，而是因为环境从来没有被记录。

\section{虚拟环境：给项目一个可控边界}

虚拟环境的作用是隔离项目依赖。它不能解决所有复现问题，但可以避免不同项目互相污染。

\begin{examplebox}[使用 \texttt{venv} 的最小流程]
\begin{lstlisting}[style=terminal]
$ python3 -m venv .venv
$ source .venv/bin/activate
$ python -m pip install numpy matplotlib pandas
$ python -m pip freeze > requirements.txt
\end{lstlisting}
\end{examplebox}

这里推荐使用 \texttt{python -m pip}，而不是单独使用 \texttt{pip}。前者明确表示“用当前这个 Python 解释器对应的 pip 安装包”，能减少装错环境的问题。

一个最小环境记录至少应该包含：

\begin{itemize}
    \item Python 版本；
    \item 关键库版本；
    \item 环境创建或激活方法；
    \item 可重建环境的依赖清单；
    \item 如果使用 conda、uv、poetry 或容器，应说明对应文件和命令。
\end{itemize}

\begin{warnbox}[不要把“我电脑能跑”当成环境说明]
如果别人不知道你用了哪个 Python、哪些库版本、从哪个目录运行脚本、输入数据放在哪里，那么“我电脑能跑”没有科研复现意义。
\end{warnbox}

\section{随机种子：固定的是过程，不是真理}

许多模拟、抽样、机器学习切分和模型训练都包含随机性。固定随机种子可以让教学示例和调试过程更稳定。

\begin{examplebox}[固定随机种子]
\begin{lstlisting}[style=pythonstyle]
import random

random.seed(42)
sample = random.sample(range(100), 5)
print(sample)
\end{lstlisting}
\end{examplebox}

但固定随机种子并不表示结论自动可靠。它只保证在相同环境下随机过程可重复。真正的科学结论还需要检查结果是否对不同随机种子、不同数据划分或不同参数选择足够稳定。这个思想会在机器学习部分反复出现。

\section{Notebook、script 与 module 的三层分工}

本章最重要的概念，是把 notebook、script 和 module 分成三层。它们不是互相替代，而是服务于不同阶段，见表~\ref{tab:ch04_notebook_script_module}。

\begin{table}[htbp]
\centering
\begin{tabular}{p{0.18\textwidth}p{0.28\textwidth}p{0.40\textwidth}}
\toprule
对象 & 最适合做什么 & 不适合承担什么 \\
\midrule
notebook & 探索、讲解、可视化、课堂演示 & 长期自动化、隐藏状态下生成最终结果 \\
script & 顺序执行、批处理、生成结果、保存日志 & 写大量解释性长文、反复复制函数 \\
module & 保存稳定函数、类和工具，被脚本或 notebook 调用 & 直接承担项目叙事或交互探索 \\
\bottomrule
\end{tabular}

\label{tab:ch04_notebook_script_module}
\end{table}

可以用一句话记住：

\begin{center}
notebook 负责解释和探索，script 负责运行和交付，module 负责复用和维护。
\end{center}

如果所有内容都堆在 notebook 中，初期看起来方便，后期往往难以检查。反过来，如果一开始就把还不稳定的想法写成复杂模块，也会降低探索效率。正确做法是随着项目成熟逐步迁移。

\section{代码成熟度：从草稿到项目入口}

同一段代码在不同阶段有不同位置。可以把成熟过程看成四级：

\begin{enumerate}
    \item \textbf{草稿单元}：在 notebook 中快速试验，允许临时变量和打印；
    \item \textbf{命名函数}：把重复逻辑改成有输入和输出的函数；
    \item \textbf{模块函数}：把稳定函数放入 \texttt{src/}，供脚本和 notebook 调用；
    \item \textbf{脚本入口}：把完整流程组织成可从命令行运行的脚本。
\end{enumerate}

不是所有草稿都要升级，也不是所有函数都要变成复杂模块。判断标准很简单：如果某段逻辑会被重复使用、被同学复查、或生成报告结果，它就不应该只停留在临时 notebook 单元里。

\section{Notebook 为什么容易制造“假可复现”}

notebook 的最大优势是交互；最大风险也来自交互。常见问题包括：

\begin{itemize}
    \item 单元格不是从上到下运行；
    \item 变量来自已经删除或修改过的旧单元；
    \item 输出图像保存在文件里，但生成代码已经变了；
    \item 中间结果依赖内存状态；
    \item notebook 中混入大量临时测试代码。
\end{itemize}

一个 notebook 如果要成为可信材料，至少应该满足：

\begin{enumerate}
    \item kernel 重启后从头运行不报错；
    \item 关键输入路径和参数在开头集中声明；
    \item 临时探索和最终结果有清楚标记；
    \item 输出文件由明确单元或脚本生成；
    \item 结尾有简短结果解释和限制说明。
\end{enumerate}

\begin{protip}[清空内核重跑是 notebook 的最低测试]
在提交 notebook 前，至少做一次 restart kernel and run all。它能暴露许多隐藏状态问题。
\end{protip}

\section{从探索性 notebook 到稳定脚本}

把 notebook 整理成脚本，不是简单复制粘贴。更好的流程是：

\begin{enumerate}
    \item 在 notebook 中完成探索，保留问题、图和解释；
    \item 标记哪些代码是最终流程，哪些只是临时试验；
    \item 把重复代码抽成函数；
    \item 把输入路径、输出路径和参数集中到开头；
    \item 把顺序稳定的执行步骤移动到脚本；
    \item 把可复用函数放进模块；
    \item 让 notebook 调用脚本输出或模块函数，而不是复制全部逻辑。
\end{enumerate}

\begin{examplebox}[探索代码到函数]
\begin{lstlisting}[style=pythonstyle]
def select_good_rows(rows):
    return [row for row in rows if row["quality"] == "good"]

def mean_magnitude(rows):
    values = [float(row["mag"]) for row in rows]
    return sum(values) / len(values)
\end{lstlisting}
\end{examplebox}

函数的好处不是“更像工程代码”，而是让每一步有名字、有输入、有输出，便于测试和复用。后续脚本可以调用这些函数，notebook 也可以调用这些函数来展示结果。

\section{函数边界：输入、输出和副作用}

整理 notebook 时，最容易犯的错误是把函数写成“偷偷依赖外部变量”。例如一个函数直接读取全局变量 \texttt{rows}，或者在函数内部悄悄写文件，却不在名字和文档里说明。更好的函数边界应回答三件事：

\begin{itemize}
    \item 输入是什么；
    \item 输出是什么；
    \item 是否有副作用，例如写文件、画图、修改全局状态。
\end{itemize}

\begin{examplebox}[边界清楚的函数]
\begin{lstlisting}[style=pythonstyle]
def select_good_rows(rows, quality_flag="good"):
    """Return rows whose quality field matches quality_flag."""
    return [row for row in rows if row["quality"] == quality_flag]


def write_summary(output_path, summary_lines):
    """Write summary lines to output_path."""
    output_path.parent.mkdir(exist_ok=True)
    output_path.write_text(
        "\n".join(summary_lines) + "\n",
        encoding="utf-8",
    )
\end{lstlisting}
\end{examplebox}

第一个函数没有副作用，只返回筛选结果；第二个函数的名字明确说明会写文件。这样的边界让脚本更容易测试，也让 notebook 更容易解释。

\section{脚本入口：为什么 \texttt{main()} 很重要}

一个脚本如果所有代码都写在顶层，导入时也会自动执行，不利于测试和复用。常见做法是把主要流程放进 \texttt{main()}：

\begin{examplebox}[最小脚本入口]
\begin{lstlisting}[style=pythonstyle]
from pathlib import Path


def main():
    project_root = Path(".")
    data_path = project_root / "data" / "lightcurve_demo.csv"
    output_path = project_root / "results" / "summary.txt"
    output_path.parent.mkdir(exist_ok=True)
    output_path.write_text(f"input={data_path}\n", encoding="utf-8")


if __name__ == "__main__":
    main()
\end{lstlisting}
\end{examplebox}

这段结构有三个好处：

\begin{itemize}
    \item 脚本可以直接运行；
    \item 其中函数可以被其他代码导入；
    \item 执行入口清楚，便于日志记录和错误定位。
\end{itemize}

\section{命令行参数：让脚本不再靠手改代码}

如果每次换输入文件都要打开脚本手动改路径，脚本就还不是真正的稳定入口。Python 标准库中的 \texttt{argparse} 可以让脚本从命令行接收参数：

\begin{examplebox}[最小参数入口]
\begin{lstlisting}[style=pythonstyle]
import argparse
from pathlib import Path


def parse_args():
    parser = argparse.ArgumentParser()
    parser.add_argument("--data", type=Path, required=True)
    parser.add_argument("--output", type=Path, required=True)
    parser.add_argument("--quality", default="good")
    return parser.parse_args()


def main():
    args = parse_args()
    args.output.parent.mkdir(exist_ok=True)
    args.output.write_text(
        f"data={args.data}\nquality={args.quality}\n",
        encoding="utf-8",
    )
\end{lstlisting}
\end{examplebox}

这样运行时就可以写：

\begin{lstlisting}[style=terminal]
$ python scripts/run_quality_check.py \
    --data data/lightcurve_demo.csv \
    --output results/summary.txt \
    --quality good
\end{lstlisting}

参数化脚本的好处是：运行命令本身就记录了输入、输出和关键选择。后续 Git 记录、日志和 README 都可以引用同一条命令。

\section{路径与配置：不要把个人机器写进代码}

科研项目中常见坏习惯是把个人电脑上的绝对路径写死：

\begin{lstlisting}[style=terminal]
/Users/student/Desktop/final_data/lightcurve.csv
\end{lstlisting}

这会让代码离开你的电脑就失效。更稳妥的方式是使用项目相对路径，并把关键路径集中管理：

\begin{examplebox}[最小配置对象]
\begin{lstlisting}[style=pythonstyle]
from dataclasses import dataclass
from pathlib import Path


@dataclass
class Config:
    data_path: Path = Path("data/lightcurve_demo.csv")
    result_path: Path = Path("results/summary.txt")
    log_path: Path = Path("logs/run_quality_check.log")
    quality_flag: str = "good"
\end{lstlisting}
\end{examplebox}

这里的配置对象很小，但它表达了一个重要思想：项目参数应该集中、显式、可审查，而不是散落在各个单元格和函数内部。

\section{日志与错误：让脚本说明自己做了什么}

脚本不是黑箱。一次可靠运行至少应留下运行时间、输入路径、输出路径和关键检查结果。最小做法可以是写一个文本日志：

\begin{examplebox}[最小日志写法]
\begin{lstlisting}[style=pythonstyle]
from datetime import datetime
from pathlib import Path

log_lines = [
    f"started={datetime.now().isoformat()}",
    "input=data/lightcurve_demo.csv",
    "output=results/summary.txt",
    "quality_flag=good",
]

Path("logs").mkdir(exist_ok=True)
Path("logs/run_quality_check.log").write_text(
    "\n".join(log_lines) + "\n",
    encoding="utf-8",
)
\end{lstlisting}
\end{examplebox}

正式项目可以使用 \texttt{logging} 模块，但入门阶段先做到“运行后有记录”更重要。没有日志的脚本，一旦结果异常，只能靠记忆排查。

\section{最小测试：先测函数，再测脚本}

本科项目不需要一开始就建立复杂测试框架，但至少应该能手动测试关键函数。比如：

\begin{examplebox}[最小函数测试]
\begin{lstlisting}[style=pythonstyle]
rows = [
    {"mag": "12.1", "quality": "good"},
    {"mag": "99.0", "quality": "bad"},
]

selected = select_good_rows(rows)
assert len(selected) == 1
assert selected[0]["mag"] == "12.1"
\end{lstlisting}
\end{examplebox}

\texttt{assert} 的意思是：如果条件不成立，就立刻报错。它不是为了证明程序永远正确，而是为了捕捉最基本的误改。对脚本则至少应做一次从头运行：

\begin{lstlisting}[style=terminal]
$ python scripts/run_quality_check.py
$ echo $?
$ ls -lh results logs
\end{lstlisting}

函数测试检查局部逻辑，脚本运行检查完整入口。两者结合，才能把 notebook 中的探索变成更可靠的科研代码。

\section{一个最小可复现脚本应交付什么}

本章的交付不应该只是“提交一个 notebook”。更合理的最小交付包括：

\begin{itemize}
    \item \texttt{README.md}：说明如何运行；
    \item \texttt{requirements.txt} 或环境说明；
    \item \texttt{notebooks/explore\_lightcurve.ipynb}：解释探索过程；
    \item \texttt{scripts/run\_quality\_check.py}：稳定运行入口；
    \item \texttt{src/lightcurve\_tools.py}：可复用函数；
    \item \texttt{logs/run\_quality\_check.log}：运行记录；
    \item \texttt{results/summary.txt}：派生结果；
    \item Git commit hash：说明这一套材料对应哪个版本。
\end{itemize}

这套交付物正好连接前两章：Linux 负责目录、脚本和日志，Git 负责版本和历史，本章负责 Python 工作流结构。

\section{配套 notebook 如何训练}

本章配套 notebook 会引导你完成一次小迁移：

\begin{enumerate}
    \item 在 notebook 中读取和检查一份小型表格；
    \item 识别哪些代码是探索，哪些代码应当稳定下来；
    \item 把重复逻辑整理成函数；
    \item 生成一个脚本草稿；
    \item 记录 Python 版本和关键依赖；
    \item 运行脚本并检查输出；
    \item 写出一份最小脚本运行记录。
\end{enumerate}

这份 notebook 的重点不是展示更多 Python 技巧，而是训练一种迁移能力：从“我在 notebook 里试出来了”走向“别人可以按照 README 和脚本重新运行”。

\section{常见错误与诊断}

\begin{warnbox}[科研编程工作流中的典型问题]
\begin{itemize}
    \item 所有代码都堆在 notebook 中，无法从头重跑；
    \item notebook 单元乱序执行，结果依赖隐藏状态；
    \item 没有环境文件，别人不知道安装哪些依赖；
    \item 写死个人电脑绝对路径；
    \item 随机过程没有固定，也没有说明稳定性检查；
    \item 脚本没有 \texttt{main()}，导入时自动执行；
    \item 重要结果没有日志、运行命令和 commit hash；
    \item 重复复制代码，而不是抽成函数或模块。
\end{itemize}
\end{warnbox}

诊断这类问题时，可以按顺序问：

\begin{enumerate}
    \item 换一台机器能否安装环境？
    \item 从项目根目录能否运行脚本？
    \item 清空 notebook 内核后能否从头运行？
    \item 输出文件是否来自明确命令？
    \item 结果是否连接到 Git 版本？
\end{enumerate}

\section{AI 助手如何帮助你}

AI 助手很适合参与本章的工作流整理。它可以帮助你：

\begin{itemize}
    \item 阅读 notebook，指出哪些单元是临时探索；
    \item 把重复代码提炼成函数；
    \item 生成脚本骨架；
    \item 建议配置对象和日志字段；
    \item 检查 README 中是否缺少运行步骤；
    \item 帮你解释环境报错。
\end{itemize}

但 AI 助手不能替你确认科学语义。例如质量筛选条件、异常值处理、单位解释和最终结论，仍然需要你根据数据和科学问题判断。一个好的提示词不是“帮我整理代码”，而是：

\begin{lstlisting}[style=terminal]
请把这段 notebook 代码按 notebook / script / module 三层分工整理。
保留探索说明，提取可复用函数，给出脚本入口，并指出哪些参数
必须写入运行记录。
\end{lstlisting}

\section{练习与交付}

\end{exercisebox}

\end{exercisebox}

\section{Exit Gate}

\begin{enumerate}
    \item notebook 和 script 的分工是否清楚？
    \item 脚本能否从命令行重跑？
    \item 环境、参数、随机种子是否记录？
    \item 输出目录是否固定且可定位？
\end{enumerate}

最低交付：Evidence Record、可运行脚本和运行日志。

\section{本章小结}

本章的核心不是让你在 notebook、script 和 module 之间选择一个“最好”的工具，而是学会分工。notebook 适合探索和解释，script 适合稳定运行和交付，module 适合保存可复用逻辑。只要你能记录环境、固定必要随机性、避免个人路径、提供脚本入口、保存日志，并把结果连接到 Git 版本，你的 Python 代码就开始从“会跑”走向“可以被复查的科研工作流”。
