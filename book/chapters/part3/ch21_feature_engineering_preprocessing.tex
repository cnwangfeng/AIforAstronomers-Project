\chapter{特征工程与数据预处理}

\section{本章导读：预处理会改变模型看见的世界}

特征工程和预处理经常被讲成模型前的清洗步骤，好像只是把数据整理得更好看。但在科学机器学习中，预处理会直接改变模型可见的信息、距离关系、权重尺度和泄漏风险。缺失值怎么填、特征如何标准化、类别变量怎么编码、哪些派生量能不能用，都会改变模型真正学习到的东西。

本章的核心不是背一组预处理技巧，而是建立一个原则：所有从数据到特征矩阵的变换都必须被记录，并且只能从训练集学习该学习的信息。否则，标准化均值、缺失值填补规则、特征选择甚至派生列，都可能把测试信息提前泄漏给模型。

读这一章时，请把预处理当成数据语义的一部分。一个特征进入模型之前，应该能回答它来自哪里、单位是什么、是否由标签间接推导、是否只用训练数据拟合，以及它改变了哪种模型假设。

\section{学习目标}

学完本章后，你应该能够：

\begin{itemize}
    \item 解释为什么同一个模型会因为特征表达不同而表现差很多；
    \item 理解缺失值处理、标准化、类别编码和特征构造各自在解决什么问题；
    \item 区分哪些特征更接近物理信息，哪些只是仪器、观测流程或数据管线留下的痕迹；
    \item 明白预处理步骤必须只在训练集上拟合，从而避免数据泄漏；
    \item 认识到特征工程不是“多造几列”，而是在重新定义模型真正看到的世界。
\end{itemize}

本章的微型项目是一份预处理审计。正文负责解释缺失值、标准化、类别编码、领域特征、特征删除和数据泄漏；训练材料则要求你把预处理步骤写成一份可以复查的 pipeline 记录。

\begin{examplebox}[最小预处理项目结构]

\section{为什么预处理会直接改变模型结果}

很多初学者会把机器学习理解成“换模型”。但在真实任务中，特征组织方式往往和模型本身同样重要。即使模型完全不变，只要你改变：

\begin{itemize}
    \item 缺失值怎么填；
    \item 数值特征是否标准化；
    \item 类别变量怎么编码；
    \item 是否构造了更贴近科学问题的新特征；
    \item 是否去掉了带有仪器痕迹的伪特征；
\end{itemize}

模型的行为就可能发生明显变化。

这说明预处理不是模型外面的“准备工作”，而是模型定义本身的一部分。你让模型看到什么、以什么尺度看到、哪些信息被强调、哪些信息被抑制，都会直接影响它学到的边界。

\section{一个教学版预处理问题}

本章配套 notebook 使用 \nolinkurl{object_preprocessing_demo.csv}。这个数据集刻意包含几类在真实项目中很常见的情况：

\begin{itemize}
    \item \textbf{缺失值}：部分样本缺少 \texttt{seeing\_arcsec} 或 \texttt{sky\_region}；
    \item \textbf{量纲差异}：颜色指数大约在 $0$ 到 $2$ 之间，而 \texttt{detector\_x} 的量级在数百像素；
    \item \textbf{类别特征}：\texttt{sky\_region} 不是连续数值，而是离散标签；
    \item \textbf{可疑特征}：\texttt{detector\_x} 更像探测器坐标，而不是天体物理属性。
\end{itemize}

这个设计的意义，是让我们看到：预处理不是“美化数据”，而是在决定模型真正把什么当成结构。

\section{缺失值：先承认，再处理}

缺失值不是异常状况，而是科学数据的常态。常见原因包括：

\begin{itemize}
    \item 某次观测失败；
    \item 某个波段没有覆盖；
    \item 数据清洗时剔除了不可靠测量；
    \item 不同巡天或仪器之间字段不完整。
\end{itemize}

如果一个模型无法接受缺失值，我们就必须先决定如何补。最简单的教学做法，通常是：

\begin{itemize}
    \item 对数值特征用训练集均值填补；
    \item 对类别特征用训练集众数填补。
\end{itemize}

这并不是唯一正确的方法，但足够帮助学生建立一个关键原则：\textbf{填补参数必须由训练集决定，而不是偷看测试集。}

\section{缺失值也可能携带信息}

缺失值并不总是“随机掉了一个数”。在观测数据里，某个字段缺失本身可能意味着：

\begin{itemize}
    \item 目标太暗，某个波段没有可靠测量；
    \item 天区没有被某次巡天覆盖；
    \item 管线因为质量问题拒绝给出结果；
    \item 某类对象更容易触发缺测。
\end{itemize}

因此，填补缺失值之前要先问：缺失机制是否和科学目标有关。如果缺失不是随机的，简单均值填补虽然能让代码跑下去，却可能抹掉非常重要的观测条件信号。实际项目中常见做法是同时加入一个缺失指示特征，例如 \texttt{is\_seeing\_missing}，让模型知道这个值曾经缺失过。

\section{为什么“先看全数据再补”会导致泄漏}

设某个数值特征在训练集上的均值为 $\mu_{\mathrm{train}}$，在测试集上的均值为 $\mu_{\mathrm{test}}$。如果你直接用全数据均值

\[
\mu_{\mathrm{all}} =
\frac{N_{\mathrm{train}}\mu_{\mathrm{train}} + N_{\mathrm{test}}\mu_{\mathrm{test}}}
{N_{\mathrm{train}} + N_{\mathrm{test}}}
\]

来填补训练数据，那么训练阶段实际上已经吸收了测试集的信息。这就是最典型的数据泄漏之一。

它危险的地方在于：代码可能完全能跑，结果甚至看起来更好，但这个“更好”不再代表真实泛化能力，而只是因为模型提前知道了一点本不该知道的信息。

\section{标准化：让距离和权重变得可比较}

对基于距离的模型、对尺度敏感的线性模型，或者任何依赖数值大小比较的算法，不同量纲都会直接改变模型的几何结构。

最常见的标准化写法是：

\[
z = \frac{x - \mu_{\mathrm{train}}}{\sigma_{\mathrm{train}}},
\]

其中 $\mu_{\mathrm{train}}$ 和 $\sigma_{\mathrm{train}}$ 都必须由训练集估计。

\begin{examplebox}[标准化的直觉]
\begin{lstlisting}[style=pythonstyle]
z = (x - mean_train) / std_train
\end{lstlisting}
\end{examplebox}

标准化的作用，不是把数据“变好看”，而是让不同特征在比较距离或系数时站到更可比的尺度上。

\section{标准化的一步数值例子}

假设训练集中某个颜色特征的均值和标准差为：

\[
\mu_{\mathrm{train}}=0.80,\qquad \sigma_{\mathrm{train}}=0.20.
\]

一个测试样本的原始颜色为 $x=1.10$，则标准化后为：

\[
z=\frac{1.10-0.80}{0.20}=1.5.
\]

注意这里使用的是训练集统计量，而不是测试集自己的均值和标准差。测试集的角色是被同一把尺子测量，而不是重新定义尺子。这个细节看似技术，却是避免泄漏的核心。

\section{为什么量纲差异会误导分类器}

在本章 notebook 中，\texttt{detector\_x} 的量级远大于颜色指数。如果我们直接把所有原始数值喂给一个最近中心分类器，那么距离往往会主要由 \texttt{detector\_x} 决定，而不是由更接近物理意义的颜色特征决定。

这说明模型并不会自动知道“哪一列更重要”。在没有标准化或特征筛选时，它只会机械地服从输入尺度本身。

\section{类别编码：让模型读懂离散信息}

\texttt{sky\_region} 这类变量不能简单当作整数编号，因为：

\begin{itemize}
    \item \texttt{north} 不应被解释成“小于” \texttt{south}；
    \item 编号之间的人为距离没有物理意义；
    \item 如果直接用整数编码，某些模型会误以为它们具有大小次序。
\end{itemize}

因此，一个常见起点是 \textbf{one-hot 编码}：把每个类别拆成一个二元列。

\begin{examplebox}[one-hot 的最小形式]
\begin{lstlisting}[style=pythonstyle]
["equator", "north", "south"]
\rightarrow
[1, 0, 0]
\end{lstlisting}
\end{examplebox}

one-hot 编码的本质，是告诉模型：“这是互斥类别，不是有序数字。”

\section{特征工程：把领域知识放进输入}

特征工程不是随意堆更多列，而是尝试把领域知识转化成更有解释力的表示。在本章示例中，我们构造了一个简单的颜色和：

\[
\texttt{color\_sum} = \texttt{u\_g} + \texttt{g\_r} + \texttt{r\_i}.
\]

它不一定是最佳特征，但足以帮助读者看到：有时一个经过整理的组合特征，比机械地把原始列全部丢进模型更清楚。

特征工程真正想回答的问题是：

\begin{itemize}
    \item 哪种表示更贴近我们关心的物理结构；
    \item 哪些列只是原始测量，哪些列已经在表达更高级的模式；
    \item 一个组合特征能否把分散信息压缩成更可区分的量。
\end{itemize}

\section{为什么有些特征应该被丢掉}

\texttt{detector\_x} 是本章故意设计出的教学“陷阱”。它在数值上很大，容易影响距离，但从科学意义上看，它更接近探测器坐标，而不是对象的天体物理属性。

这提醒我们：不是所有“表里现成可用的列”都值得保留。特征选择并不只是统计操作，也是在问：

\begin{itemize}
    \item 这个特征是否真的携带科学信号；
    \item 它会不会只是仪器、管线或观测条件的痕迹；
    \item 它会不会导致模型学到“表面高分、实际错误”的规律。
\end{itemize}

\section{预处理改变的，不只是数值，而是边界}

本章 notebook 用一个最小最近中心分类器做三组对照：

\begin{itemize}
    \item 只做最小填补，直接使用原始数值列；
    \item 对数值特征做标准化；
    \item 进一步做特征工程、类别编码，并去掉明显更像仪器坐标的特征。
\end{itemize}

这三组对照的价值，不在于追求某个绝对最优分数，而在于让读者看到：\textbf{同一个模型，只要输入表示改变，决策边界就会改变。}

\section{Pipeline 与 ColumnTransformer 为什么重要}

当预处理步骤变多时，手工操作很容易出错。例如：

\begin{itemize}
    \item 先填补、后标准化，还是先标准化、后填补；
    \item 哪些列做 one-hot，哪些列做数值缩放；
    \item 测试集是不是误用了全数据的均值；
    \item 新增特征是否和训练时保持完全一致。
\end{itemize}

这正是 \texttt{Pipeline} 和 \texttt{ColumnTransformer} 的价值所在：它们能把预处理和模型训练明确串起来，减少手工重复和泄漏风险。

\section{把预处理写成训练/应用两阶段算法}

无论是否使用标准库，一个可靠的预处理流程都应分成两个阶段：

\begin{enumerate}
    \item \textbf{fit 阶段}：只在训练集上学习均值、方差、众数、编码表和特征选择规则；
    \item \textbf{transform 阶段}：把已经学到的规则应用到训练集、验证集、测试集和未来新数据。
\end{enumerate}

\begin{examplebox}[预处理算法骨架]
\begin{lstlisting}[style=pythonstyle]
preprocessor = fit_preprocessor(train_rows)

train_features = transform(preprocessor, train_rows)
test_features = transform(preprocessor, test_rows)

model = train_model(train_features, train_labels)
predictions = model.predict(test_features)
\end{lstlisting}
\end{examplebox}

这段骨架的关键在于：测试集永远不能调用 \texttt{fit\_preprocessor}。它只能使用训练阶段已经固定好的转换规则。很多数据泄漏并不发生在模型训练函数里，而是发生在这一步之前。

\section{一个最小预处理审计表}

预处理最容易出问题的地方，是“代码看起来只是整理数据”，但实际上已经改变了模型能看到的信息。因此，每一次正式实验都应留下一个预处理审计表。它不需要很长，但必须能回答：

\begin{enumerate}
    \item \textbf{哪些列进入模型}：原始列、工程特征和被删除列分别是什么；
    \item \textbf{统计量从哪里来}：均值、方差、众数、编码表和特征选择规则是否只由训练集拟合；
    \item \textbf{转换顺序是什么}：先填补、再标准化、再编码，还是存在其他顺序；
    \item \textbf{测试集如何处理}：测试集只调用已经拟合好的转换器，而不是重新估计参数；
    \item \textbf{疑似伪特征如何处理}：仪器坐标、观测批次、质量标记等列是否被单独审查。
\end{enumerate}

\begin{examplebox}[预处理审计表的最小字段]
\begin{lstlisting}[style=pythonstyle]
preprocessing_audit = {
    "numeric_features": ["u_g", "g_r", "r_i", "seeing_arcsec"],
    "categorical_features": ["sky_region"],
    "engineered_features": ["color_sum"],
    "removed_features": ["detector_x"],
    "fit_only_on_train": True,
    "leakage_checks": ["imputer", "scaler", "encoder", "feature_selection"],
}
\end{lstlisting}
\end{examplebox}

这个审计表的价值，不只是防止自己忘记做过什么；更重要的是让别人能判断：模型得到的改进究竟来自更合理的物理表示，还是来自不小心把测试信息、仪器痕迹或数据管线偏差喂给了模型。

在最终项目里，预处理审计表主要进入 Data card、Reproducibility card 和 Integrity card。它回答的是三个朴素问题：模型到底看见了哪些列，这些转换能不能复跑，以及有没有把不该看的测试信息或伪特征带进去。

\section{最需要警惕的不是技术错误，而是数据泄漏}

预处理里最隐蔽的风险之一，就是在划分训练集和测试集之前，就先对全数据做：

\begin{itemize}
    \item 均值填补；
    \item 标准化；
    \item 特征选择；
    \item 类别编码规则拟合；
    \item 组合特征统计量估计。
\end{itemize}

这样一来，测试集的信息就提前泄漏给了模型。哪怕模型结构本身没变，评估结果也会被污染。

\begin{warnbox}[预处理中最常见的误区]
\begin{itemize}
    \item 用全数据的均值填补缺失值；
    \item 忽视量纲差异，让大尺度噪声特征主导模型；
    \item 把类别变量直接编码成有序整数；
    \item 机械增加特征数量，而不判断它们是否有物理意义；
    \item 在没有清楚记录步骤的情况下手工反复处理数据。
\end{itemize}
\end{warnbox}

\section{AI 助手如何帮助你}

在这一章，AI 助手适合帮助你：

\begin{itemize}
    \item 检查你的缺失值处理和标准化代码是否只使用了训练集统计量；
    \item 讨论某个特征更像物理信号还是仪器痕迹；
    \item 帮你把手写预处理流程整理成更清楚的函数或 \texttt{Pipeline} 结构；
    \item 协助你把“为什么丢掉某一列”写成更明确的科学理由。
\end{itemize}

但哪些特征真正值得保留、是否存在隐蔽泄漏、某个工程特征是否合理，仍然必须由你结合科学背景判断。

\section{本章小结}

特征工程与数据预处理不是模型之外的“准备工作”，而是模型定义本身的一部分。你让模型看到什么、以什么尺度看到、缺失值如何补、类别如何编码、哪些特征被保留或删除，都会直接决定它学到的规律是否可信。本章通过一个教学版预处理问题，把缺失值、标准化、编码、特征工程和数据泄漏放进同一条工作流里，为后面更复杂的模型选择与泛化问题打好了基础。
