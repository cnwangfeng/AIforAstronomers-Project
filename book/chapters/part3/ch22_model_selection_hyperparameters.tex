\chapter{模型选择与超参数}

\section{本章导读：调参是受控比较，不是追逐最高分}

模型选择看起来像是在寻找“最好的模型”，但更准确地说，它是在一组候选假设之间做受控比较。每一次选择模型复杂度、正则化强度、邻居数、树深度或聚类数量，本质上都在改变模型能够表达的结构，也在改变过拟合、欠拟合和外推风险。

本章强调验证集和交叉验证，是为了防止学生把测试集变成反复调参的工具。测试集一旦参与选择，就不再是独立证据。网格搜索也不是魔法；它只是把候选模型系统排列出来，再用事先规定的评估规则比较。

读这一章时，请把每一次超参数选择都写成记录：候选范围是什么，baseline 在哪里，验证分数是否稳定，最终模型为什么值得被保留，测试集何时才被打开。这样调参才是科研流程，而不是试运气。

\section{学习目标}

学完本章后，你应该能够：

\begin{itemize}
    \item 区分模型参数与超参数，并理解两者在训练流程中的职责完全不同；
    \item 解释为什么模型选择必须依赖验证集或交叉验证，而不能直接偷看测试集；
    \item 理解欠拟合、过拟合与模型复杂度之间的关系，而不是把“更复杂”误认为“更先进”；
    \item 用一个清楚的工作流完成候选模型比较、超参数选择、最终重训练与独立测试；
    \item 知道在数据量有限时，为什么交叉验证往往比单次验证切分更稳健。
\end{itemize}

\section{参数和超参数不是一回事}

很多学生第一次进入模型选择时，会把所有“可调东西”混在一起。实际上，至少要区分两类量：

\begin{itemize}
    \item \textbf{参数}：由训练过程根据数据学出来，例如线性回归系数、神经网络权重、决策树节点阈值；
    \item \textbf{超参数}：由研究者在训练外部指定，例如多项式阶数、决策树深度、$k$ 近邻中的 $k$、正则化强度等。
\end{itemize}

参数回答的是“模型如何贴合这一批训练数据”；超参数回答的是“我们允许模型具有什么复杂度和归纳偏好”。

\begin{defbox}[模型选择的核心对象]
模型选择通常不是在直接比较某一组已训练参数，而是在比较不同超参数设定、不同复杂度甚至不同模型族对应的泛化能力。
\end{defbox}

\section{为什么模型选择是一个独立问题}

只要你尝试了很多候选模型或很多超参数组合，你做的事情就不再只是“训练一个模型”，而是在进行一轮\textbf{选择}。一旦有选择，就必须追问：

\begin{itemize}
    \item 你根据什么标准做出了选择；
    \item 这个标准是否已经偷看了测试集；
    \item 你最终报告的分数，是否已经被选择过程污染。
\end{itemize}

这就是为什么真实工作流里通常要把数据划分成：

\begin{itemize}
    \item \textbf{训练集}：拟合模型参数；
    \item \textbf{验证集}：比较候选模型与超参数；
    \item \textbf{测试集}：在所有选择都冻结后，只做最后一次独立报告。
\end{itemize}

\section{复杂度太低会欠拟合，太高会过拟合}

超参数往往直接控制模型复杂度。例如：

\begin{itemize}
    \item 多项式回归的阶数越高，函数越灵活；
    \item 决策树最大深度越大，边界越复杂；
    \item $k$ 近邻里的 $k$ 越小，模型越追随局部细节；
    \item 正则化强度越弱，模型越容易记住训练集细枝末节。
\end{itemize}

如果复杂度太低，模型连主要趋势都抓不住，会出现\textbf{欠拟合}；如果复杂度太高，模型会把噪声也当成规律，会出现\textbf{过拟合}。模型选择的任务，就是在这两者之间找到更合适的平衡，而不是机械追求更复杂的函数形式。

\section{一张典型验证曲线应该怎样读}

模型选择最常见的一类图，是横轴为复杂度、纵轴为误差的验证曲线。它通常会出现三种区域：

\begin{itemize}
    \item \textbf{低复杂度区域}：训练误差和验证误差都高，说明模型欠拟合；
    \item \textbf{适中复杂度区域}：训练误差较低，验证误差也较低，说明模型抓住了主要结构；
    \item \textbf{高复杂度区域}：训练误差继续下降，但验证误差回升，说明模型开始过拟合。
\end{itemize}

这张图的教学价值，是让学生看到“更复杂”并不是单调变好。真正要选的，不是训练误差最低的点，而是验证误差低且相对稳定的区域。

\section{一个教学版颜色--温度回归例子}

本章配套 notebook 使用 \nolinkurl{stellar_teff_model_selection_demo.csv}，它是一个小型 Gaia 风格教学数据集，包含：

\begin{itemize}
    \item 颜色指标 \texttt{bp\_rp}；
    \item 参考有效温度 \texttt{teff\_k}。
\end{itemize}

这个例子之所以适合作为模型选择入门，是因为它同时满足：

\begin{itemize}
    \item 颜色与温度之间确实存在明显但非线性的关系；
    \item 候选模型可以用\textbf{多项式阶数}这一单一超参数来控制复杂度；
    \item 一次验证切分和交叉验证会给出略有不同的选择结果，足以说明“稳定性”问题。
\end{itemize}

\section{单次验证集能做什么，不能做什么}

在 notebook 中，我们先设定一组候选阶数：

\begin{examplebox}[候选超参数]
\begin{lstlisting}[style=pythonstyle]
degrees = [1, 2, 3, 5]
\end{lstlisting}
\end{examplebox}

随后先做一次固定训练/验证切分，比较每个阶数的训练集 RMSE 与验证集 RMSE。这个步骤能帮我们快速排除明显不合适的候选：

\begin{itemize}
    \item 一次项模型可能过于简单，训练和验证误差都偏高；
    \item 五次多项式可能在训练集上误差很低，但验证误差明显恶化；
    \item 二次和三次模型则更可能处于“可用候选”区域。
\end{itemize}

单次验证切分的价值在于直观，但它的局限也很明显：结果可能高度依赖于这一轮样本分配。

\section{为什么交叉验证是在给选择过程降噪}

如果数据量不大，单次验证切分的结果常会对样本选择非常敏感。设交叉验证一共有 $K$ 个折，则最简单的平均验证误差写作：

\[
\mathrm{CV}(\lambda) =
\frac{1}{K}\sum_{k=1}^{K} E_k(\lambda),
\]

其中 $\lambda$ 表示某个超参数配置，$E_k(\lambda)$ 表示第 $k$ 折上的验证误差。

这个平均值并不是“更神秘的分数”，它只是在回答一个更稳健的问题：如果换几种合理切分方式，这个超参数还会不会表现得不错。

因此，交叉验证的真正意义是：

\begin{itemize}
    \item 让选择不至于依赖某一次幸运切分；
    \item 估计超参数表现是否稳定；
    \item 让我们看到“平均效果”和“波动幅度”。
\end{itemize}

\section{交叉验证的最小算法}

把 $K$ 折交叉验证写成伪代码，就是：

\begin{examplebox}[K 折交叉验证骨架]
\begin{lstlisting}[style=pythonstyle]
scores = []
for fold in folds:
    fold_train = all_rows_except(fold)
    fold_valid = fold

    model = train_with_hyperparameter(fold_train, lambda_value)
    predictions = model.predict(fold_valid)
    scores.append(metric(fold_valid, predictions))

mean_score = mean(scores)
score_spread = standard_deviation(scores)
\end{lstlisting}
\end{examplebox}

这里有两个结果都要保留：平均分和波动幅度。平均分告诉你“通常表现怎样”，波动幅度告诉你“这个表现是否稳定”。若两个候选平均分差别很小，但其中一个波动明显更小，在科研项目里后者往往更值得信任。

\section{训练误差下降，不代表模型更好}

模型选择中最值得反复提醒的一点，是：\textbf{训练误差持续下降，不意味着泛化能力持续提高。}

随着多项式阶数升高，模型几乎总能在训练集上更好地贴合数据；但在验证集或交叉验证上，它可能已经开始追随噪声。也就是说，训练集看到的是“拟合能力”，验证集看到的更接近“泛化能力”。

\section{测试集为什么必须最后才出现}

完成超参数选择之后，我们再用训练集与验证集的合并数据重新拟合最终模型，并且只在这时看一次测试集结果。这个步骤的逻辑非常关键：

\begin{itemize}
    \item 训练集和验证集负责\textbf{选择}；
    \item 测试集负责\textbf{报告}。
\end{itemize}

如果你在测试集上反复比较不同阶数，再挑一个最好看的结果写进报告，那么测试集就已经不再是测试集，而是被你偷偷变成了验证集。

\begin{protip}[测试集最重要的属性是独立性]
测试集并不是“最重要所以最常看”的那部分数据；恰恰相反，它之所以重要，是因为它应该在选择结束前尽量不被看见。
\end{protip}

\section{一个值得记住的标准工作流}

\begin{enumerate}
    \item 先定义科学问题和评价指标；
    \item 划分训练、验证、测试，或在训练阶段使用交叉验证；
    \item 明确候选模型与超参数范围；
    \item 只用训练/验证信息完成选择；
    \item 冻结方案后，用训练加验证数据重新拟合最终模型；
    \item 最后只在测试集上评估一次。
\end{enumerate}

这套流程看起来慢一点，但它是在保护最终结论的可信度。

\section{最终重训练为什么不是偷看测试集}

模型选择完成后，常见做法是把训练集和验证集合并，重新训练最终模型，然后只在测试集上评估一次。这个步骤容易让学生困惑：既然验证集已经参与选择，为什么还能合并进训练？

原因在于，选择已经结束，超参数和模型族已经冻结。此时验证集不再用于继续挑模型，而是作为额外训练样本，帮助最终模型更充分地估计参数。真正不能做的是：看完测试集结果后，又返回去修改超参数、特征、阈值或预处理。

因此，最终重训练的前提是：

\begin{itemize}
    \item 候选空间已经关闭；
    \item 最终模型配置已经写入记录；
    \item 测试集结果只报告一次；
    \item 如果测试集表现不佳，只能诚实报告，而不能继续把它用于选择。
\end{itemize}

\section{一个最小模型选择记录模板}

模型选择最怕的不是“没有找到最高分”，而是后来连自己都说不清楚为什么选择了这一版模型。为了让选择过程可复查，实验记录中至少应保留：

\begin{enumerate}
    \item \textbf{候选空间}：比较了哪些模型族、哪些超参数范围；
    \item \textbf{选择指标}：用什么指标排序，以及它为什么匹配科学目标；
    \item \textbf{验证方式}：单次验证切分、$K$ 折交叉验证，还是时间/空间分块验证；
    \item \textbf{选择结果}：最佳配置是什么，它和次优配置差多少；
    \item \textbf{最终报告}：冻结选择后，最终测试集分数是多少，是否只报告了一次。
\end{enumerate}

\begin{examplebox}[模型选择记录的最小骨架]
\begin{lstlisting}[style=pythonstyle]
selection_record = {
    "candidate_degrees": [1, 2, 3, 5],
    "selection_metric": "validation_rmse",
    "validation_design": "5-fold cross-validation on training split",
    "chosen_model": {"family": "polynomial_regression", "degree": 2},
    "runner_up": {"degree": 3, "mean_cv_rmse": 154.2},
    "final_test_used_once": True,
}
\end{lstlisting}
\end{examplebox}

如果某个模型只是比次优方案好一点点，而交叉验证波动又很大，那么报告时就不应把它写成“明显胜出”。这类细节恰恰体现科研训练与排行榜训练的差别：我们不只是挑最高分，还要说明选择是否稳定、是否值得相信。

这份选择记录后面会进入 Model card 与 Evaluation card。它让读者看见模型不是“凭感觉挑出来的”，而是在固定候选空间、固定验证设计和一次性最终测试的约束下被选中的。

\section{网格搜索不是魔法，只是系统地比较候选}

很多库把模型选择封装成 \texttt{GridSearchCV} 或类似接口，但它做的事情本质上并不神秘：

\begin{enumerate}
    \item 列出候选超参数组合；
    \item 对每个组合做训练和验证；
    \item 根据选定指标排序；
    \item 选出最优配置。
\end{enumerate}

真正困难的并不是会不会调用某个接口，而是：

\begin{itemize}
    \item 你为什么选这个指标；
    \item 候选空间是否合理；
    \item 是否把预处理也纳入了选择流程；
    \item 是否避免了数据泄漏和测试集污染。
\end{itemize}

\section{搜索太多也会制造“看上去更好”的假象}

超参数搜索并不是越大越好。如果你在很小的数据集上试几十种模型、几百组超参数，最后总能挑出一个看起来特别漂亮的组合。但这未必说明你找到了更好的规律，也可能只是把噪声筛选成了“最佳模型”。

这也是为什么模型选择必须写清楚：

\begin{itemize}
    \item 一共试了哪些候选；
    \item 选择依据是什么；
    \item 结果在不同切分上是否稳定；
    \item 最终汇报是否与选择过程分离。
\end{itemize}

\section{为什么这一章属于“科研可靠性”，不只是“调参技巧”}

很多人把超参数选择理解成一种竞赛式技巧，仿佛只是为了把分数再榨高一点。但在科研训练里，本章真正关心的不是“分数多高”，而是：

\begin{itemize}
    \item 你报告的效果是否可信；
    \item 你的选择是否稳定；
    \item 你是否能把复杂度与科学问题匹配起来；
    \item 你有没有把测试集当成了一次性证据，而不是可反复操纵的资源。
\end{itemize}

\begin{warnbox}[模型选择中的常见误区]
\begin{itemize}
    \item 用测试集挑超参数；
    \item 只看训练误差，不看验证误差；
    \item 一次切分结果很好，就认定模型已经稳定；
    \item 搜索范围极大，却没有解释这些候选为何值得比较；
    \item 在预处理、特征选择和模型比较之间反复偷看测试集。
\end{itemize}
\end{warnbox}

\section{AI 助手如何帮助你}

在这一章，AI 助手适合帮助你：

\begin{itemize}
    \item 把训练、验证、交叉验证和测试的职责分清楚；
    \item 检查你是否不小心让测试集参与了选择；
    \item 帮你整理候选超参数表格、交叉验证记录与最终模型选择说明；
    \item 协助你比较“单次验证最好”和“交叉验证最稳”之间的差别。
\end{itemize}

但候选空间为何这样设、复杂度是否和科学问题匹配、最终该报告哪一版结果，这些判断仍然必须由你自己负责。

\section{练习}

\begin{exercisebox}[基础练习]
解释为什么“训练误差最低的模型”通常不是应该被直接报告的最终模型。
\end{exercisebox}

\begin{exercisebox}[进阶练习]
从交叉验证平均误差的定义出发，说明它为什么能比单次验证切分更稳健，但仍然不能替代最终测试集。
\end{exercisebox}

\begin{exercisebox}[开放练习]
为一个你感兴趣的回归或分类任务写一份模型选择方案，至少包含：候选超参数、选择指标、验证方式、最终测试策略，以及你最担心的“测试集污染”风险。
\end{exercisebox}

\section{本章小结}

模型选择不是训练结束后的附带步骤，而是决定结果是否可信的核心流程。超参数控制模型复杂度；验证集和交叉验证帮助你做选择；测试集只负责最后的独立检验。把这三件事分开，是从“会跑模型”走向“会做可靠科学推断”的关键一步。本章通过颜色--温度回归这个最小示例，把复杂度、验证、交叉验证和测试集职责串成了一条完整而可复现的科学工作流。
